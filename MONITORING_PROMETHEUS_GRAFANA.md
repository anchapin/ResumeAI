# Prometheus + Grafana Monitoring Stack (Issue #418)

Comprehensive production-ready monitoring setup for Resume API with Prometheus metrics collection, Grafana dashboards, and AlertManager alerting.

## Overview

This implementation provides:

- **Prometheus**: Time-series metrics database with scraping and storage
- **Grafana**: Visualization dashboards for metrics
- **AlertManager**: Alert routing and notifications
- **Custom Metrics Exporter**: Application-specific metrics collection
- **Metrics Middleware**: Auto-instrumentation of HTTP endpoints
- **Alert Rules**: Pre-configured alerts for critical metrics
- **Recording Rules**: Pre-computed metrics for dashboard performance

## Quick Start

### Option 1: Docker Compose (Recommended)

```bash
# Start the full monitoring stack
docker-compose -f docker-compose-monitoring.yml up -d

# Check status
docker-compose -f docker-compose-monitoring.yml ps

# View logs
docker-compose -f docker-compose-monitoring.yml logs -f prometheus
docker-compose -f docker-compose-monitoring.yml logs -f grafana
docker-compose -f docker-compose-monitoring.yml logs -f resume-api

# Stop the stack
docker-compose -f docker-compose-monitoring.yml down
```

### Metrics Flow

1. **Collection**: Resume API exports metrics in Prometheus format at `/metrics`
2. **Scraping**: Prometheus scrapes metrics every 15 seconds (configurable)
3. **Storage**: Metrics stored in Prometheus TSDB with 30-day retention
4. **Computation**: Recording rules pre-compute aggregations for dashboard queries
5. **Evaluation**: Alert rules evaluated every 30 seconds
6. **Routing**: Triggered alerts sent to AlertManager
7. **Notification**: AlertManager routes to appropriate channels (Slack, Email, PagerDuty)
8. **Visualization**: Grafana queries Prometheus for dashboard display

## Metrics Categories

### HTTP Request Metrics

**Counters:**
- `http_requests_total` - Total HTTP requests by method/endpoint/status
- `http_errors_total` - Total HTTP errors
- `rate_limit_exceeded_total` - Rate limit violations

**Histograms:**
- `http_request_duration_seconds` - Request latency
- `http_request_size_bytes` - Request body size
- `http_response_size_bytes` - Response body size

**Gauges:**
- `http_requests_in_progress` - Requests currently being processed

### Business Metrics

**Counters:**
- `pdfs_generated_total` - PDFs generated by variant
- `resumes_tailored_total` - Resumes tailored by AI provider
- `variants_listed_total` - Variant list operations

**Histograms:**
- `pdf_generation_duration_seconds` - PDF generation latency
- `resume_tailor_duration_seconds` - Resume tailor latency

**Gauges:**
- `active_resumes` - Number of active resumes
- `total_users` - Total registered users

### AI Provider Metrics

**Counters:**
- `ai_requests_total` - Total AI API requests
- `ai_request_tokens_total` - Tokens used (input/output)
- `ai_request_cost_usd` - Cost of AI requests
- `ai_rate_limit_hits_total` - Rate limit hits
- `ai_provider_errors_total` - Provider errors

**Histograms:**
- `ai_request_duration_seconds` - AI API latency

### Database Metrics

**Counters:**
- `db_queries_total` - Total queries by operation/table
- `db_slow_queries_total` - Queries exceeding 1 second

**Histograms:**
- `db_query_duration_seconds` - Query latency
- `db_transaction_duration_seconds` - Transaction latency

**Gauges:**
- `db_connections_active` - Active connections
- `db_connections_waiting` - Waiting connections

### Cache Metrics

**Counters:**
- `cache_hits_total` - Cache hits
- `cache_misses_total` - Cache misses
- `cache_evictions_total` - Cache evictions

**Gauges:**
- `cache_size_bytes` - Cache size

### Queue/Job Metrics

- `async_job_retries_total` - Job retries

**Histograms:**
- `async_job_duration_seconds` - Job processing time

**Gauges:**
- `async_queue_depth` - Current queue depth

### OAuth Metrics

**Counters:**

## Alert Rules

### Alert Tiers

**CRITICAL (Immediate Response)**
- Service Down
- High Error Rate (>5%)
- Low PDF Generation Success (<90%)
- High OAuth Failure Rate (>5%)

**WARNING (Investigate)**
- High Response Time (P99 > 5s)
- High Request Rate (>1000 req/sec)
- Client Error Spike (>100/sec)
- Queue Backlog (>100 jobs)
- Slow DB Queries (P99 > 2s)
- Low Cache Hit Rate (<70%)

### Recording Rules

Pre-computed aggregations for faster dashboard queries:

**HTTP Metrics:**
- `http:requests:rate5m` - Request rate by endpoint
- `http:latency:p50/p95/p99` - Latency percentiles
- `http:error_rate:percentage` - Error rate %

**Business Metrics:**
- `business:pdfs:rate5m` - PDF generation rate
- `business:tailors:rate5m` - Resume tailor rate

**Database Metrics:**
- `db:latency:p50/p95/p99` - Query latency percentiles
- `db:slow_queries:count` - Slow query count

**Cache Metrics:**
- `cache:hit_rate:percentage` - Cache hit rate %
- `cache:hits:rate5m` - Hit rate
- `cache:misses:rate5m` - Miss rate

## Configuration Files

### prometheus.yml

Main Prometheus configuration:
- Scrape intervals (15s default)
- Job definitions (resume-api, node-exporter, etc.)
- Relabel configs for label enrichment
- Alert and recording rule files

**Key Sections:**
```yaml
global:
  scrape_interval: 15s
  evaluation_interval: 15s

alerting:
  alertmanagers:
    - targets: ['localhost:9093']

rule_files:
  - alert_rules.yml
  - recording_rules.yml

scrape_configs:
  - job_name: 'resume-api'
    static_configs:
      - targets: ['localhost:8000']
```

### alert_rules.yml

Alert definitions with conditions and notifications:
- API Performance alerts
- Business Metrics alerts
- Queue & Job alerts
- Resource alerts
- Database alerts
- Cache alerts
- OAuth alerts
- AI Provider alerts

**Alert Structure:**
```yaml
groups:
  - name: group_name
    rules:
      - alert: AlertName
        expr: metric_query
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Human-readable summary"
          description: "Detailed description"
          runbook: "Link to runbook"
```

### recording_rules.yml

Pre-computed metrics for dashboard optimization:
- HTTP metrics aggregations
- Business metrics roll-ups
- Database performance metrics
- Cache efficiency metrics
- Resource utilization summaries

### alertmanager.yml

Alert routing and notification configuration:
- Route definitions by severity/component
- Receiver configurations (Slack, Email, PagerDuty)
- Inhibition rules (suppress related alerts)
- Mute time intervals (maintenance windows)

## Integration with Resume API

### 1. Middleware Installation

Add metrics middleware to FastAPI app:

```python
from middleware.metrics_middleware import MetricsMiddleware

# Add to middleware stack (after other middleware)
app.add_middleware(MetricsMiddleware)
```

### 2. Metrics Endpoint

Expose metrics at `/metrics`:

```python
from prometheus_fastapi_instrumentator import Instrumentator

instrumentator = Instrumentator(
    excluded_handlers=["/metrics"],
)
instrumentator.instrument(app).expose(
    app,
    endpoint="/metrics",
    should_gzip=True,
)
```

### 3. Custom Metrics

Use PrometheusExporter for custom metrics:

```python
from lib.monitoring.prometheus_exporter import get_exporter

exporter = get_exporter()

# Record HTTP request
exporter.record_http_request(
    method="POST",
    endpoint="/v1/render/pdf",
    status_code=200,
    duration=0.45,
    request_size=5000,
    response_size=250000
)

# Record business event
exporter.record_pdf_generation(
    variant="modern",
    template="standard",
    duration=2.5,
    status="success"
)

# Record AI request
exporter.record_ai_request(
    provider="openai",
    model="gpt-4",
    duration=1.2,
    input_tokens=150,
    output_tokens=500,
    cost=0.025,
    status="success"
)
```

- Evaluation interval: 30 seconds
- Retention: 7 days for detailed metrics, 30 days for summaries

- Pre-computed recording rules

- Caching enabled (60s default)

## Troubleshooting

### Prometheus Not Scraping

**Problem**: Targets show "Down" in Prometheus UI

**Solutions:**
1. Check Resume API is running: `curl http://localhost:8000/health`
2. Verify metrics endpoint: `curl http://localhost:8000/metrics`
3. Check Prometheus logs: `docker-compose logs prometheus`
4. Verify prometheus.yml syntax: `promtool check config prometheus.yml`

### No Metrics in Grafana

**Problem**: Dashboard panels show "No data"

**Solutions:**
1. Verify Prometheus has data: Query in Prometheus UI
2. Check scrape_interval: Data points should exist every 15s
3. Verify datasource connection in Grafana
4. Check time range selector on dashboard

### Alerts Not Firing

**Problem**: Expected alerts not triggering

**Solutions:**
1. Verify alert rule syntax: `promtool check rules alert_rules.yml`
2. Check alert evaluation: Prometheus Alerts tab
3. Verify AlertManager is running
4. Check AlertManager logs: `docker-compose logs alertmanager`

### High Memory Usage

**Problem**: Prometheus using excessive memory

**Solutions:**
1. Reduce `scrape_interval` or `evaluation_interval`
2. Decrease retention time: `--storage.tsdb.retention.time=7d`
3. Reduce cardinality: Drop high-cardinality labels
4. Increase container memory limit

## Best Practices

### 1. Metric Naming

Follow Prometheus naming conventions:
- Use `_total` suffix for counters: `pdfs_generated_total`
- Use `_seconds` suffix for durations: `http_request_duration_seconds`
- Use `_bytes` suffix for sizes: `memory_usage_bytes`
- Describe metric in first line of help text

### 2. Label Design

- Minimize label cardinality (low values per metric)
- Use consistent label names across metrics
- Avoid unbounded labels (e.g., user_id)
- Include important dimensions: method, endpoint, status_code

### 3. Alert Design

- Write clear, actionable alert descriptions
- Include links to runbooks for resolution
- Use appropriate `for` clause (prevent flapping)
- Group related alerts with routes

### 4. Dashboard Design

- Use consistent color schemes
- Add descriptions to panels
- Include time range selectors
- Use auto-refresh (30s recommended)
